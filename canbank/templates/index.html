IndyKite Banking

Use Knowledge-Based Access Control (KBAC) and ContX IQ (CIQ) to retrieve or upsert CanBank data according to context. CIQ provides fine-grained, context-aware data access across the CanBank organization: departments, employees, customers, accounts, policy documents, regulatory agreements and decisions.

How ContX IQ Works
1. Policy

Defines what data can be accessed and who can access it. Uses Cypher patterns to match graph structures and filters to bind parameters. Specifies allowed reads, upserts, and deletes.

2. Knowledge Query

Defines what data to return from the matched graph. Specifies which node properties and relationships to include in results.

3. Execute

Runs the query with input parameters. The policy validates access, the query retrieves data, and results are returned. Uses the application's credentials for authorization.

Data Capture

Import the CanBank dataset (data/nodes/nodes_banking.json and data/relationships/relationships_banking.json) into the IndyKite Knowledge Graph (IKG). Creates the CanBank organization hierarchy, employees, customers, accounts, policy documents, regulatory agreements, tickets and decisions — plus the relationships connecting them.

Capture Nodes

Import entities: Entity, Department, User, Customer, Account, AccountTier, Document, Folder, RegulatoryAgreement, InvestmentCategory, Jurisdiction, Ticket, Decision, Quote, System
Capture Relationships

Import connections: CONTAINS, WORKS_IN, REPORTS_TO, OWNS, CAN_ACCESS, CAN_RETRIEVE, IS_TYPE, IS_TIER, CLASSIFIED_AS, SUBJECT_TO, ENFORCED_IN, CREATED_BY, REGARDING, CLOSED_BY, MADE, EXCEPTION_TO

ContX IQ Use Cases
1 Get Self
Policy Query Execute

Return the authenticated user's profile together with their department and (optionally) their manager.

Reads: user, department, manager
Graph Pattern: (subject:User)-[:WORKS_IN]->(:Department), (subject)-[:REPORTS_TO]->(:User)
Input: bearer token only
2 Get Stock Quote
Policy Query Execute

Return a stock quote when the caller's department is allowed to retrieve it (e.g. Trading).

Reads: user, department, quote
Graph Pattern: (:User)-[:WORKS_IN]->(:Department)-[:CAN_RETRIEVE]->(:Quote)
Input: ticker
3 Get Stock Trade Threshold
Policy Query Execute

Return a customer's stock account tier and threshold. Only reachable from users in the Trading department.

Reads: user, customer, stock account, tier
Graph Pattern: (:User)-[:WORKS_IN]->(:Department)-[:CAN_ACCESS|CONTAINS*..2]->(:Customer)-[:OWNS]->(:Account)-[:IS_TIER]->(:AccountTier)
Input: customer_external_id
4 Get Internal Documents
Policy Query Execute

Return internal documents reachable from the caller's department, filtered by a taxonomy (e.g. policy).

Reads: user, document
Graph Pattern: (:User)-[:WORKS_IN|CAN_ACCESS|CONTAINS*..4]->(:Document)-[:IS_TYPE]->(:Taxonomy)
Input: taxonomy_external_id
5 Get Customer Facing Documents
Policy Query Execute

Return customer facing documents together with their regulatory agreements and investment categories.

Reads: document, regulatory agreement, investment category
Graph Pattern: (:Document)-[:SUBJECT_TO]->(:RegulatoryAgreement), (:Document)-[:CLASSIFIED_AS]->(:InvestmentCategory)
Input: none
6 Get Regulatory Agreements
Policy Query Execute

Return the full list of regulatory agreements known to the graph.

Reads: regulatory agreement name + description
Graph Pattern: (:RegulatoryAgreement)
Input: none
7 Get Decisions
Policy Query Execute

For a given document, return the decisions that referenced it along with the ticket, customer and account.

Reads: user, document, decision, ticket, account, customer
Graph Pattern: (:Document)<-[:EXCEPTION_TO]-(:Decision)<-[:CLOSED_BY]-(:Ticket)-[:REGARDING]->(:Account), (:Ticket)-[:CREATED_BY]->(:Customer)
Input: document_external_id
